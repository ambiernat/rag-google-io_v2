в—Ҹ Here's the design review focused on agentic retrieval readiness:

  ---
  Design Review: Agentic Retrieval Readiness

  What's solid

  - Clean, consistent retriever signatures вҖ” all three return List[ScoredPoint]
  with identical (query, top_k, client) parameters. Strategy switching is
  already mechanical.
  - Metrics module is importable вҖ” retrieval/evaluation/metrics.py functions
  work standalone, no script-level coupling.
  - Reranker has model caching вҖ” safe to call inside a loop without reload
  overhead.
  - experiment_id on SearchRequest вҖ” A/B tracking is already wired.
  - Dev/val/test split is locked and reproducible вҖ” evaluation discipline is
  sound.

  ---
  Design gaps that will hurt the agentic build

  1. No unified retriever dispatcher

  The agentic loop (Phase 3) needs strategy switching. Right now there's no
  single retrieve(query, strategy, top_k) entry point вҖ” the caller must branch
  manually every time. This logic will be duplicated across the agent, any
  evaluation harness, and any future caller.

  Recommendation: Add retrieval/retrievers/dispatcher.py:
  def retrieve(query: str, strategy: Literal["dense","sparse","hybrid"], top_k: 
  int) -> List[ScoredPoint]:
      ...

  ---
  2. Impedance mismatch: ScoredPoint вҶ’ dict conversion is scattered

  Retrievers return ScoredPoint. The reranker takes list[dict] with a "text"
  key. The API converts inline. The evaluation scripts convert inline. The
  agentic layer will convert inline again вҖ” each time duplicating
  .payload.get("doc_id", str(hit.id)).

  Recommendation: Add a single conversion utility:
  def scored_points_to_dicts(hits: List[ScoredPoint]) -> List[dict]:
      ...
  Place it in retrieval/retrievers/ or a new retrieval/utils.py. Every caller
  uses it, nothing converts inline.

  ---
  3. No RetrievalResult dataclass

  ScoredPoint is a Qdrant implementation detail. The agentic layer will work
  with text, scores, and doc IDs вҖ” it shouldn't care about Qdrant internals.
  Passing raw ScoredPoint objects into an LLM-calling agent is a leaky
  abstraction.

  Recommendation: Define a thin domain type:
  @dataclass
  class RetrievalResult:
      doc_id: str
      score: float
      text: str
      title: str | None = None
      video_id: str | None = None
  Retrievers can still return ScoredPoint internally; convert at the boundary
  before handing to the agent.

  ---
  4. Evaluation scripts are not importable

  evaluate_dense.py, evaluate_sparse.py, evaluate_hybrid.py all use if __name__ 
  == "__main__" with hardcoded paths and no function boundary. An agentic
  evaluation loop needs to call evaluation programmatically, not via subprocess.

  The logic is simple but the structure blocks reuse. Phase 4 (final evaluation)
   will require restructuring these anyway.

  Recommendation: Extract run_evaluation(gt_path, strategy, top_k) -> dict from
  each script, or create a single evaluate_retrieval(strategy, gt_path, top_k)
  callable. Scripts become thin wrappers.

  ---
  5. No deduplication/merge utility for multi-query results

  Phase 1 (query rewriter) generates 2вҖ“3 query variants and retrieves on each.
  You'll get overlapping ScoredPoint lists that need merging and dedup before
  passing to the self-evaluator. No utility for this exists.

  RRF (Reciprocal Rank Fusion) is already in Qdrant's hybrid config вҖ” but for
  merging results from multiple queries in Python, you'd need your own
  implementation.

  Recommendation: Add merge_results(results: List[List[RetrievalResult]], k: 
  int) -> List[RetrievalResult] with RRF or score-max fusion.

  ---
  6. No designated location for the agentic component

  The project structure has retrieval/retrievers/, retrieval/rerankers/,
  retrieval/hpo/ вҖ” but no retrieval/agent/ or top-level agent/. Where does
  query_rewriter.py, self_evaluator.py, and the retry loop live?

  Recommendation: Create retrieval/agent/ with:
  retrieval/agent/
  в”ңв”Җв”Җ __init__.py
  в”ңв”Җв”Җ query_rewriter.py      # Phase 1
  в”ңв”Җв”Җ self_evaluator.py      # Phase 2
  в”ңв”Җв”Җ retry_loop.py          # Phase 3
  в””в”Җв”Җ config.yaml            # agent-specific config (model, thresholds)

  This keeps the agentic logic co-located with the retrieval layer it
  orchestrates, not mixed into api/ or evaluation/.
